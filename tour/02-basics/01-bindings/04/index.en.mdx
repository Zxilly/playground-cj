# Type Inference

When a variable has an initial value, the compiler can automatically infer the variable's type based on the initial value's type, without explicit annotation. This makes code more concise while maintaining type safety. Type inference is one of the features of Cangjie's type system.

In the code on the right, all variables lack type annotations, but the compiler knows their types: integer literals are inferred as `Int64`, floating-point literals as `Float64`, string literals as `String`, and boolean literals as `Bool`. These are Cangjie's default type inference rules. Values computed through expressions also have their types inferred.

Type inference is not dynamic typing—the variable's type is determined at compile time and doesn't change at runtime. If you try to assign a string to a variable inferred as an integer, or assign a floating-point number to a variable inferred as an integer, the compiler will report an error. Cangjie is a statically typed language, and type checking occurs at compile time.

<Note>While type inference is convenient, explicitly annotating types is still valuable in certain cases: improving code readability, avoiding unexpected type inference, or helping the compiler when inference isn't precise enough.</Note>

<CompareWith lang="java">

<CompareWith lang="go">
Go also supports type inference (via `:=` syntax), where integer literals are inferred as `int` (typically 64-bit) and floating-point numbers as `float64`, similar to Cangjie's default inference rules.
</CompareWith>

<CompareWith lang="rust">
Rust also supports type inference, but integer literals are inferred as `i32` (32-bit integer) by default, while Cangjie defaults to `Int64` (64-bit integer), which is more suitable for modern 64-bit systems.
</CompareWith>
