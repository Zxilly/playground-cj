# Minimizing Critical Sections

When using mutexes, an important principle is to minimize the scope of critical sections. The larger the critical section, the longer other threads have to wait, resulting in worse concurrency performance.

A good pattern is: first complete time-consuming calculations locally, and only hold the lock for the brief moment when shared data needs to be accessed. This can significantly reduce lock contention and improve the program's concurrency performance.

In the code on the right, each worker thread constructs a message string locally (a relatively time-consuming operation) and only enters the `synchronized` block when adding a message to the shared list. This is much more efficient than holding the lock during the entire message construction period.

The output will show messages from different workers, and the order may vary due to concurrent scheduling.

<Note>If the critical section is too small (e.g., only protecting partial shared operations), it may not properly protect data. The key is to ensure that complete operations on shared data are within the critical section.</Note>

Try moving message construction inside the `synchronized` block - while the result will be the same, performance will decrease.

CJ:
import std.sync.Mutex
import std.collection.ArrayList

let messages = ArrayList<String>()
let messagesLock = Mutex()

main(): Int64 {
    let futures = ArrayList<Future<Unit>>()
    
    // Start 5 worker threads
    for (workerId in 1..=5) {
        let current = workerId
        let future = spawn {
            // Perform time-consuming calculations outside critical section
            let message = "Worker ${current} completed task"
            
            // Simulate some computational work
            var sum = 0
            for (i in 0..1000) {
                sum += i
            }
            
            // Only hold lock when accessing shared data
            synchronized(messagesLock) {
                messages.add(message)
            }
        }
        futures.add(future)
    }
    
    // Wait for all threads to complete
    for (future in futures) {
        future.get()
    }
    
    // Print all messages
    println("Received ${messages.size} messages:")
    for (message in messages) {
        println("  - ${message}")
    }
    
    return 0
}
