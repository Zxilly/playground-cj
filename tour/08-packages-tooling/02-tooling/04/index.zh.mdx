# 工作区（Workspace）

当一个项目包含多个模块时，可以使用工作区（Workspace）来统一管理。

在根目录的 `cjpm.toml` 中使用 `[workspace]` 字段，并通过 `members` 指定成员模块列表。每个成员模块都有自己的 `cjpm.toml` 配置文件。

工作区的目录结构通常是：根目录有一个工作区配置文件，每个子目录是一个独立的模块。在工作区根目录运行 `cjpm build` 会编译所有成员模块。

<Note>`[workspace]` 和 `[package]` 不能在同一个 `cjpm.toml` 中共存。工作区配置文件只定义成员，各模块的具体配置在各自的 `cjpm.toml` 中。</Note>

查看代码注释中的目录结构示例，理解工作区的组织方式。

<CompareGroup>

<CompareWith lang="c">
C 没有工作区的概念。大型项目通常通过 Make 或 CMake 配置多个子项目的编译过程。
</CompareWith>

<CompareWith lang="java">
Java 的 Maven 或 Gradle 支持多模块项目，在根项目的 `pom.xml` 或 `build.gradle` 中使用 `<modules>` 或 `include()` 定义子模块。
</CompareWith>

<CompareWith lang="go">
Go 通过 `go.work` 文件定义工作区，使用 `directory` 指定模块目录。仓颉用 `cjpm.toml` 中的 `[workspace]` 字段管理。
</CompareWith>

<CompareWith lang="rust">
Rust 使用 `Cargo.toml` 中的 `[workspace]` 字段，与仓颉的工作区配置几乎完全一致。
</CompareWith>

</CompareGroup>
