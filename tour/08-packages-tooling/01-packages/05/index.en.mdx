# Re-exports

`public import` can re-export imported declarations, making them visible to code that imports your package. This is called "re-exporting."

Default `import` is `private`, meaning imported names are only available in the current file. With `public import`, these names become part of your package's public API.

Re-exports are commonly used to build convenient APIs for libraries. Library authors can aggregate commonly used types from multiple subpackages into a single entry package, so users only need to import that one package. This simplifies usage while preserving flexibility in internal organization.

<CompareWith lang="c">
C has no concept of modules or packages, making re-exports impossible. All imports are handled directly through preprocessor directives.
</CompareWith>

<CompareWith lang="java">
Java does not support re-exports. Imported classes are only available in the current file. To provide a unified API, you need to create wrapper classes in an intermediate package.
</CompareWith>

<CompareWith lang="go">
Go doesn't support re-exports. To provide simplified imports, you can only write wrapper types or functions manually. Cangjie's `public import` is more direct and convenient.
</CompareWith>

<CompareWith lang="rust">
Rust supports re-exports: `pub use std::collections::HashMap;`. The mechanism is similar to Cangjie's, both used to expose internal dependencies or simplify APIs for users.
</CompareWith>

</CompareGroup>
