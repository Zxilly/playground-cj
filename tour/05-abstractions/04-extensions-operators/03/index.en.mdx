# Access Rules

Extensions are located outside the original type definition and must comply with access control rules. Extensions can only access `public` and `protected` members of the extended type; they cannot access `private` fields and methods.

The `Rectangle` struct in the code on the right has a private `_tag` field and public `width`, `height` fields. Functions defined in the extension like `area()`, `perimeter()`, `isSquare()` can only calculate through public fields and cannot access `_tag`.

Extensions cannot shadow existing members of the extended type, nor can they shadow non-`private` members defined in other extensions. This ensures deterministic type behavior.

<CompareGroup>

<CompareWith lang="c">
C has no explicit access control and relies on file scope and conventions; the concept of extensions doesn't exist.
</CompareWith>

<CompareWith lang="java">
Java methods can access private fields through reflection (with special code), while Cangjie enforces access control directly at the compiler level during the extension phase.
</CompareWith>

<CompareWith lang="go">
Go only has capitalization conventions for visibility; extensions (if they existed) could only access members starting with capital letters.
</CompareWith>

<CompareWith lang="rust">
Rust trait implementations can access all fields (within the same module), subject to visibility constraints across modules.
</CompareWith>

</CompareGroup>

<Note>Extensions themselves cannot use access modifiers, but extension members can use modifiers like `public`, `private`, `protected`.</Note>

Try accessing private fields in the extension and observe the compiler's error messages.
