# Type Constraints

The generic parameter `T` can represent any type, but this creates a problem: inside the function body, the compiler doesn't know the concrete type of `T`, so it cannot call any of its member functions or access properties. For example, you cannot perform `+ 1` on a value of type `T` or print it.

The `where` clause can constrain generic parameters, specifying interfaces they must implement or parent classes they must inherit from. For instance, `where T <: ToString` means `T` must implement the `ToString` interface, allowing you to safely call the `toString()` method inside the function body.

The code on the right defines a `Printable` interface, then uses `where T <: Printable` to constrain the generic functions `printItem` and `printAll`. Inside these functions, you can call the `display()` method because the compiler knows that all passed types implement this interface.

<CompareGroup>

<CompareWith lang="c">
C has no generics or constraint mechanisms; you must use preprocessor macros or manually implement multiple versions of functions without type safety guarantees.
</CompareWith>

<CompareWith lang="java">
Java uses `<T extends SomeClass>` or `<T extends Interface>` to constrain generic parameters, but type erasure causes runtime information to be lost.
</CompareWith>

<CompareWith lang="go">
Go's generic constraints use `any` or interface types, written in square brackets after the function parameter list, such as `[T Stringer]`.
</CompareWith>

<CompareWith lang="rust">
Rust uses trait bounds to constrain generics, written as `<T: Display>` or `where T: Display`.
</CompareWith>

</CompareGroup>

<Note>Without specifying a constraint, generic parameters can only perform basic operations like assignment and passing, and cannot call any methods.</Note>
