# Invariance of Generic Types

Although `Dog` is a subtype of `Animal`, `Container<Dog>` does not automatically become a subtype of `Container<Animal>`. This is because in Cangjie, user-defined generic types are **invariant** at their type parameters.

This design avoids type safety issues. For example, if `Array<Dog>` could be used as `Array<Animal>`, you could add a `Cat` to it, breaking type safety.

There are two ways to solve polymorphism problems: using constraints on generic parameters, or using base type arrays to store different subtypes. The code on the right demonstrates both approaches: the `makeSpeak` function uses `where T <: Animal` constraint and can accept any `Animal` subtype; the `speakAll` function uses `Array<Animal>` and can store mixed subtypes.

<CompareGroup>

<CompareWith lang="c">
C has no concept of subtypes or generics; manual type conversion is required and errors are easy to make.
</CompareWith>

<CompareWith lang="java">
Java's generics support covariant wildcards `<? extends T>` and contravariant wildcards `<? super T>`, which are flexible but also complex; Cangjie's invariance is simpler.
</CompareWith>

<CompareWith lang="go">
Go's generics are also invariant, but similar effects can be achieved through slices of interface types.
</CompareWith>

<CompareWith lang="rust">
Rust's generics are also invariant by default, using trait bounds to solve polymorphism problems, with similar logic.
</CompareWith>

</CompareGroup>

<Note>Built-in tuple types are covariant at their element types, and function types are contravariant at their parameter types and covariant at their return types.</Note>
